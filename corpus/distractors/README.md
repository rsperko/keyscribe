# Distractor corpus — recognition-bias / fuzzy-recovery false-fire measurement

Ordinary sentences whose audio is **acoustically adjacent** to a dictionary term, with a realistic
small dictionary (10 neutral tech terms) active on every clip. On a **distractor** clip the reference
contains **no** dictionary term, so any term the engine emits is a **false fire**. **Control** clips do
speak a term, so term recall confirms bias/recovery still fires when it should — a threshold that kills
every fire would otherwise look perfect.

Two probe classes cover the failure modes:

- **Multi-word phrases** whose join is acoustically near a compound term ("text field" → TextField,
  "code base" → CodeBase) — the split-merge class.
- **Single-token near-misses** one edit from a short term but phonetically distinct ("lava" vs Java,
  "dust" vs Rust) — the classic edit-distance false-positive band the fuzzy phonetic gate must reject —
  plus long-word (norm ≥ 7) distance-3 probes ("grafting"/"graffiti" vs Grafana) that must stay blocked.

This is the harm-side complement to `corpus/stt/` (which is term-dense, so it can only see the
*benefit* of bias). Built to answer: at shipped thresholds, how often does decode-time recognition
bias replace a common phrase with a sound-alike dictionary term?

## Layout

- `manifest.json` (committed) — 34 distractors + 10 controls, each rendered in 3 `say` accents
  (Samantha US, Daniel UK, Karen AU) = 132 synthetic clips. `checks.stt.biasTerms` = the full 10-term
  dictionary on every clip; `tags` carry `distractor`/`control` + the target term.
- `<id>.wav` (gitignored) — 16 kHz mono, regenerated by `gen.py`.
- `gen.py` — renders the wavs via `say` + `afconvert` and (re)writes the manifest, preserving any
  human takes. Neutral public terms only (repo hygiene).

## Run

```bash
./make-app.sh                                               # bundles the metallib Qwen needs
./KeyScribeDev.app/Contents/MacOS/KeyScribe --benchmark corpus/distractors
```

The table reports `ff(bias)` / `ff(unbias)` — total false fires (bias terms present in the
hypothesis but absent from the reference). The **unbiased** arm is the engine-hallucination
baseline; spotter/prompt/context-caused fires are the **delta** over it. Recall on the control
clips confirms bias still fires when a term is genuinely spoken.

## Caveats

- **TTS is clean acoustics**, so it likely *under*-measures false fires vs a real mic in a real
  room. Human takes slot in as additional `manifest.json` entries (record with
  `corpus/record.sh`) — the metric reads them the same way.
- False-fire and recall both use **case-insensitive substring containment** (matching
  `BenchmarkScoring.termRecall`). Distractor references are written so no dictionary term is an
  accidental substring; keep that property when adding clips.
- Two terms are CamelCase compounds (`TextField`, `CodeBase`): `say` pronounces them as two words,
  so their control clips specifically test whether bias *produces* the CamelCase form — recall 0 on
  those means bias did not fire, not that the audio was wrong.
- **Substring scoring has a morphological blind spot.** A false fire needs the term absent from the
  reference, tested by case-insensitive substring. A word that *contains* a term ("rusty" ⊃ "rust")
  therefore can't be a distractor here — the reference would count as mentioning the term, masking a
  real snap. So a suffix-inflection probe like "a rusty old truck" → Rust is intentionally excluded;
  measuring it needs a token-exact metric, not substring containment.
